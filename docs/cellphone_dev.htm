Developer Notes

 

The following document serves to give a brief overview of the file structure and design of the Cellphone web server, and is intended for those who might improve it in the future. The architecture outlined in this overview is not the only suitable architecture, and can be modified if wanted.

 

Also if you’re wondering why this is a PDF rather than a text file, I didn’t want to deal with differences in line break format between Windows and *nix systems in case you like to edit using a Windows program rather than vim in a Cygwin terminal.

 

Knowledge Requirements:

 

Cellphone uses some part of all of the following technologies:

 

·      Sinatra: A lightweight web framework for Ruby, used to write the main server and the histogrammer server

·      HAML: A form of HTML that is easier to write (less brackets and tags)

·      JSON: A simple format for transferring structured data over the internet

·      JQuery: A JavaScript library used to develop responsive web pages

·      Unsemantic: A CSS library used for formatting the look of web pages

·      Highcharts.js: A JavaScript library used to display charts within a webpage

 

Directories and Files:

 

The Cellphone project is organized in a directory structure based on Sinatra and Ruby on Rails conventions.

 

·      Gemfile (and Gemfile.lock): A list of all gems required by the system

·      app.rb: Main server application file, loads helpers, routes, and other modules on startup

·      app.ru: Main server rack configuration file

·      config.yml: Main server configuration file, used to set some immutable global variables

·      helpers: Directory containing all custom modules used within the main application server

o   imagecapture.rb: Module containing image capture methods

o   init.rb: File loaded by app.rb, loads all other modules in helpers directory

o   javascripts.rb: Module contains methods for including Javascript files in Sinatra routes, rather than hard coding in HAML files

o   logs.rb: Module used to log experiment data and events to files in logs directory

o   package.rb: Module used to package experiment results into zipfile

o   partials.rb: Module used to include small pieces of HAML within large HAML files

o   styles.rb: Module contains methods for including CSS files in Sinatra routes, rather than hard coding in HAML files

·      histogrammer.rb: Histogrammer server application file, sole purpose is to generate histograms of images

·      histogrammer.ru: HIstogrammer server rack configuration file

·      lib: Directory containing any native binaries used in either server

o   TWAINCom.exe: Program used to capture images from the microscope.

·      logs: Directory containing all log files generated when an experiment is run.

·      public: Directory containing all resources that can be requested by the client

o   assets: Directory containing all resources required by the client for the display of various web pages

§  imgs: Directory containing all images captured during the experiment

§  javascripts: Directory  containing all JavaScript files used by the client. See requests in the routes directory to determine which files are requested for each page

§  stylesheets: Directory  containing all JCSS files used by the client. See requests in the routes directory to determine which files are requested for each page

o   results: Directory containing zipfiles of all results, where a new zipfile is created at the end of an experiment.

·      routes: Directory containing all routes accessible on the server

o   api.rb: Ruby file containing all routes useful for requesting pieces of data from the server

o   experiment.rb: Ruby file containing all routes required to display various webpages from the server, and the route starting the worker thread that generates and processes images.

o   init.rb: File loaded by app.rb, loads all other routes in the routes directory

·      start_app.sh: Convenience bash script for starting the main application server

·      start_histogrammer.sh: Convenience bash script for starting the image histogram server

·      views: Directory contains all HAML files used to generate the HTML displayed by the client.

o   experiment: Directory containing all HAML unique to each visible page of the system

o   layout.haml: HAML file containing all boilerplate HAML required by all pages

 

Main Processes:

 

The main application server performs two main tasks:

o   Respond to requests from the client(s) with appropriate data and resources

o   Run a thread that captures images and performs experiment logic determining when doses should or should not be administered

 

The histogrammer application server performs a single task:

o   Generate a histogram for a given image

 

The histogrammer application had to be moved to a separate server because the image-processing library used within does not play nicely with threads. Moving that logic to a new server allows the library to work along side all processes occurring in the main application server without breaking the library.

 

Next Steps

 

The Cellphone system is usable but could be greatly improved. The following is a partial list of some possible avenues for future work:

 

o   Write an open source TWAIN interface in C that can capture images from the microscope, removing dependencies on the TWAINCom.exe program, which is proprietary and slow. This is likely a project in itself.

o   Improve site design, its functional but not the prettiest or easiest to use.

o   Write a simple .exe program that allows users to simply click to start the entire system, rather than forcing them to use Cygwin.

o   Make the installation process simple and easy, rather than the multistep behemoth outlined in the companion cellphone_setup.pdf file.

o   Encapsulate experiment specific logic into modules that can be swapped in and out on a per-experiment basis, so that the system is useful for a variety of experiments.
